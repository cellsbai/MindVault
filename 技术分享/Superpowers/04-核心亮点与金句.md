# Superpowers — 核心亮点与金句

> 这里汇集了最适合在技术分享中重点讲解的内容，以及原文中最有力的表述。

---

## 🏆 三个最值得深讲的亮点

### 亮点 1：技能描述即触发器——一个反直觉的设计

**背景**：
大多数人写文档会把流程写在描述里，但 Superpowers 明确禁止这样做。

**原因（有真实 case）**：
有人在技能描述里写了"tasks 之间做 code review"，结果 Agent 只读了描述，只做了一次 Review，而不是正文里要求的两阶段 Review。

**设计规则**：
> 描述只写触发条件（`Use when...`），工作流程只放在正文。

**为什么有趣**：
这是一个把"AI 阅读行为"作为设计约束的例子——不是写给人看，是写给 Agent 执行的，要针对 Agent 的行为模式设计。

---

### 亮点 2：并行 Agent 分发——真正的工程效率

**传统方式**：
```
失败1 → 修复1 → 失败2 → 修复2 → 失败3 → 修复3
总时间 = t1 + t2 + t3
```

**Superpowers 方式**：
```
失败1 ──────────────────→ 修复1
失败2 ──────────────────→ 修复2  （同时进行）
失败3 ──────────────────→ 修复3
总时间 = max(t1, t2, t3)
```

**实测效果**：6 个失败用例，3 个文件，并行解决，零冲突。

**关键洞察**：给每个 Agent 的 prompt 要足够具体，"Fix agent-tool-abort.test.ts" 远好于 "Fix all the tests"。

---

### 亮点 3：诚实验证原则——对 AI 幻觉的系统性解药

**问题**：AI Agent 最容易犯的错是"幻觉式完成"——说完成了，但其实没有。

**Superpowers 的解法**：
不允许没有运行证据的"完成声明"。

**验证流程**：
```
1. 找到能证明断言的命令
2. 完整运行它
3. 检查输出和退出码
4. 确认输出支持你的声明
5. 带证据声明完成
```

**核心金句**：
> "Bypassing verification is dishonesty disguised as efficiency."  
> 绕过验证，是以效率为名的不诚实。

## 💬 原文金句集

### 关于 TDD

> "If you didn't watch the test fail, you don't know if it tests the right thing."  
> 如果你没有亲眼看到测试失败，你就不知道这个测试是否在测正确的东西。

---

### 关于调试

> "NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST"  
> 找到根因之前，禁止提出任何修复。

---

### 关于验证

> "Bypassing verification is dishonesty disguised as efficiency."  
> 绕过验证，是以效率为名的不诚实。

---

### 关于 Brainstorming 的约束

> "Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it."  
> 在你呈现设计并获得用户批准之前，不得调用任何实现技能、写任何代码、搭建任何项目框架或采取任何实现行动。

---

### 关于并行 Agent

> "Fix agent-tool-abort.test.ts" vastly outperforms "Fix all the tests."  
> 具体的指令远胜于模糊的指令。

---

### 关于技能文档

> "NO SKILL WITHOUT A FAILING TEST FIRST"  
> 没有先观察到失败，不写技能文档。

## 🎯 分享结语建议

Superpowers 告诉我们一件事：

**AI 不缺能力，缺的是约束。**

真正让 AI 编程可靠的，不是更大的模型，而是更好的工作流框架——就像优秀的工程师不只靠聪明才智，更靠良好的工程习惯。
