# Superpowers — 14 个技能模块详解

> Superpowers 的核心是 14 个可组合的技能（Skills），按职责分为四大类。

## 分类概览

```
14 个技能
├── 🧠 思考与规划（2个）
│   ├── brainstorming
│   └── writing-plans
│
├── ⚙️ 执行与协作（4个）
│   ├── subagent-driven-development
│   ├── dispatching-parallel-agents
│   ├── using-git-worktrees
│   └── executing-plans
│
├── ✅ 质量保障（5个）
│   ├── test-driven-development
│   ├── systematic-debugging
│   ├── verification-before-completion
│   ├── requesting-code-review
│   └── receiving-code-review
│
└── 🛠 元技能（3个）
    ├── writing-skills
    ├── using-superpowers
    └── finishing-a-development-branch
```

---

## 🧠 一、思考与规划

### brainstorming — 需求澄清与设计

**触发时机**：用户提出新功能或改造需求时

**九步工作流**：
1. 探索项目上下文
2. 询问是否需要视觉辅助
3. **逐一**提出澄清问题（不连炮提问）
4. 提出 2-3 个方案 + 权衡分析
5. 分段呈现设计
6. 写入设计文档（`docs/superpowers/specs/`）
7. 自我审查文档（消除歧义/占位符/矛盾）
8. 等待用户审批
9. 跳转 `writing-plans`

**最终态**：只允许跳转到 `writing-plans`，不能跳其他实现技能。

---

### writing-plans — 制定实现计划

**触发时机**：设计已批准，开始拆解实现

**核心标准**：假设工程师**零上下文**也能独立执行

每个任务必须包含：
- ✅ 精确文件路径（create / modify / test）
- ✅ TDD 顺序步骤（测试先写）
- ✅ 完整代码块（非伪代码）
- ✅ 具体命令和预期输出
- ✅ 任务后提交指令

**文件组织哲学**：
> "Split by responsibility, not by technical layer."  
> 按职责拆分，不按技术层（controller/service/repo）拆分。

**交付前自检**：
- 消除所有占位符
- 确保类型/方法在所有任务间一致
- 验证对 spec 的完整覆盖

---

## ⚙️ 二、执行与协作

### subagent-driven-development — 串行 Agent 执行

**模式**：每个任务 → 派一个全新 Agent → 两阶段 Review → 下一任务

**两阶段 Review**：
1. **规范合规 Review**：代码是否符合计划规格？
2. **代码质量 Review**：实现是否足够健壮？

**关键规则**：
- 永远不跳过 Review
- Reviewer 有顾虑时不强行通过
- 遇到 Blocking 问题，调整方案而非重试

---

### dispatching-parallel-agents — 并行 Agent 分发

**适用场景**：
- 3+ 个测试文件失败，各自根因不同
- 多个子系统同时出问题，互相独立

**判断标准**：
- ✅ 问题互不相关（修一个不影响另一个）
- ✅ 每个调查不依赖共享状态
- ❌ 失败相互关联时不适用

**每个 Agent 的信息包**：
- 聚焦范围（一个测试文件 / 一个子系统）
- 自包含上下文（不依赖外部信息）
- 明确约束（哪些不能改）
- 具体交付物（期望输出什么）

**实测效果**：6 个失败用例，3 个文件，并行解决，零冲突。

---

### using-git-worktrees — Git 隔离工作空间

**核心价值**：
- 每个功能分支有独立的物理目录
- 可以同时在多个分支工作，互不干扰
- 开发前先验证测试基线（起点干净）

---

### executing-plans — 执行计划

**两种执行模式**：
1. **Subagent 模式**：每任务一个新 Agent，上下文最干净
2. **内联模式**：当前 Agent 批量执行，定期设置检查点

---

## ✅ 三、质量保障

### test-driven-development — 测试驱动开发

**铁律**：Red → Green → Refactor

| 步骤 | 操作 | 目的 |
|------|------|------|
| RED | 写失败的测试 | 证明测试真的在测目标行为 |
| GREEN | 写最小实现通过测试 | 避免过度工程 |
| REFACTOR | 清理代码，不改功能 | 保持代码健康 |

**"看到失败"是核心**：
> "If you didn't watch the test fail, you don't know if it tests the right thing."

**常见逃避借口（全部无效）**：
- "代码太简单了"
- "我先写代码，然后补测试"
- "我手动测过了"
- "先保留代码作参考"

---

### systematic-debugging — 系统化调试

**四阶段根因分析**：

```
Phase 1：根因调查
   └── 复现问题 → 查报错 → 检查最近变更 → 追踪数据流

Phase 2：模式分析
   └── 找到能正常工作的类似代码 → 对比差异 → 理解依赖

Phase 3：假设与验证
   └── 明确因果理论 → 最小化改动测试 → 验证结果

Phase 4：实现
   └── 先写失败测试 → 修复根因 → 验证
```

**关键阈值**：连续 3 次修复失败 → 停止追症状，审查架构。

---

### verification-before-completion — 完成前验证

**规则**：声称完成之前，必须运行真实验证命令并附上输出。

**验证流程**：
1. 找到能证明断言的验证命令
2. 完整执行
3. 检查输出和退出码
4. 确认输出支持你的断言
5. 附带证据声明完成

**禁止用语**：
- "应该没问题"（should work）
- "看起来对了"（seems correct）
- "太棒了，完成！"（before running verification）

---

### requesting-code-review / receiving-code-review

**请求 Review 前检查**：
- 所有 CI 通过
- 合并冲突已解决
- 分支已与目标分支同步

**Review 严重级别**：
- 🔴 CRITICAL → 必须修复，不得合并
- 🟠 HIGH → 应修复
- 🟡 MEDIUM → 考虑修复
- ⚪ LOW → 可选

---

## 🛠 四、元技能

### writing-skills — 技能编写方法

**将 TDD 应用于文档编写本身**：

- **RED**：不用技能运行场景，记录 Agent 的失败行为
- **GREEN**：写解决具体失败的最小技能
- **REFACTOR**：迭代测试，堵住所有合理化逃避漏洞

**CSO 原则（Critical Skill Object）**：
> 描述只写触发条件（"Use when..."），不写工作流程。  
> 原因：写了流程，Agent 会只读描述不读正文。

---

### using-superpowers — 系统入口

告知 Agent 如何查找和使用技能系统，是整个框架的引导技能。

---

### finishing-a-development-branch — 完成分支

- 确认所有任务完成 + 测试通过
- 处理完所有 Review 意见
- 做出合并决策
- 清理 worktree
