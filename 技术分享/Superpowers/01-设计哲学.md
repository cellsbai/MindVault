# Superpowers — 设计哲学

> 核心问题：为什么 AI Agent 需要一套工作流框架，而不是直接写代码？

## 问题起点

AI 编程 Agent 的常见失控路径：

```
用户说"帮我做 X"
→ Agent 立刻开始写代码
→ 写了一半发现方向不对
→ 推倒重来 / 代码越修越乱
→ 用户失去信心
```

Superpowers 的回答是：**给 Agent 装上和优秀工程师一样的工作习惯。**

---

## 五大核心原则

### 1. 先理解，再动手（Gate Before Code）

> "Do NOT write any code until you have presented a design and the user has approved it."

- 没有用户确认设计文档，禁止任何实现行为
- 先提问 → 给方案 → 写设计文档 → 获批 → 才写代码
- 这是硬性门禁，不是建议

### 2. YAGNI — 你不会需要它

> You Aren't Gonna Need It

- 不为"未来可能的需求"预先设计
- 每个功能必须有当前明确的理由
- 简单的系统比复杂的系统更容易维护和验证

### 3. 诚实验证原则（Verification Honesty）

> "Bypassing verification is dishonesty disguised as efficiency."

- 任何"完成了"的声明必须附带**真实运行**的验证证据
- 禁止使用"应该没问题"、"看起来对了"等模糊表述
- 声称完成 ≠ 完成，运行证明 = 完成

### 4. 根因优先（Root Cause First）

> "NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST"

- 禁止在找到根本原因之前提出修复方案
- 症状驱动的修复会制造新问题
- 连续 3 次修复失败 → 强制反思架构层面

### 5. 技能触发即读全文（Trigger = Read Full Skill）

> 一个隐藏但关键的设计决策：

- 技能描述（description）只写**触发条件**，不写工作流程
- 原因：Agent 会只读描述而跳过正文
- 例：描述写了"tasks 间做 code review"→ Agent 只做了一次，而非两阶段

---

## 框架定位对比

| 维度 | 传统 AI 编程助手 | Superpowers |
|------|----------------|-------------|
| 入口 | 直接写代码 | 先 Brainstorm + Plan |
| 质量保障 | 靠 prompt 约束 | 结构化技能 + 强制检查点 |
| 测试 | 事后补测 | TDD 强制 Red 先失败 |
| 完成判定 | Agent 自己说完了 | 必须有验证证据 |
| 多任务 | 顺序执行 | 支持并行 Agent 分发 |

---

## 一句话总结

> Superpowers 本质是：把"优秀工程师的工作习惯"编码成 AI 可执行的技能库。
