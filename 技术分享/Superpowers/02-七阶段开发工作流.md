# Superpowers — 七阶段开发工作流

> Superpowers 将一个完整功能的开发周期分为 7 个强制阶段，每个阶段都有明确的进入条件和退出检查点。

## 全景图

```
用户想法
   │
   ▼
① Brainstorming     ── 澄清需求，提供设计方案，用户确认
   │
   ▼
② Git Worktrees     ── 创建隔离工作空间，验证测试基线
   │
   ▼
③ Writing Plans     ── 拆解为 2-5 分钟任务，含完整代码和路径
   │
   ▼
④ Subagent 执行     ── 每任务派新 Agent，两阶段 Review
   │
   ▼
⑤ Test-Driven Dev   ── Red → Green → Refactor 强制循环
   │
   ▼
⑥ Code Review       ── 按严重级别分类，规范合规 + 代码质量
   │
   ▼
⑦ Branch Completion ── 合并决策与清理
   │
   ▼
  完成
```

---

## 各阶段详解

### ① Brainstorming — 需求澄清

**技能**：`brainstorming`

**流程**：
1. 探索项目现有上下文（读文件/文档）
2. 逐一提问（单次一个问题，不连炮）
3. 提出 2-3 个备选方案及权衡
4. 分段呈现设计方案
5. 写入设计文档 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
6. 自我 Review 设计文档（查占位符/矛盾/歧义）
7. **等待用户批准** ← 强制检查点

**关键约束**：未获批准前，不得调用任何实现技能。

---

### ② Git Worktrees — 隔离工作空间

**技能**：`using-git-worktrees`

**目的**：
- 每个功能/修复在独立 worktree 中进行
- 主分支保持干净，可随时切换
- 开发前验证测试基线（确认起点没有已有失败）

---

### ③ Writing Plans — 制定计划

**技能**：`writing-plans`

**核心要求**：
- 每个任务 **2-5 分钟**可完成
- 假设执行者**零上下文**：必须包含完整代码、精确文件路径、具体命令
- 禁用占位符（`<your-logic-here>` 这类写法不允许）
- 每个任务遵循 TDD 模式：测试文件 → 实现文件

**文件组织原则**：
> "Files that change together should live together. Split by responsibility, not by technical layer."

---

### ④ Subagent 执行 — 并行/串行分发

**技能**：`subagent-driven-development` / `dispatching-parallel-agents`

**模式一：串行（Subagent-Driven）**
- 每个任务派一个全新 Agent（避免上下文污染）
- Agent 完成后进行两阶段 Review：
  1. **规范合规 Review**：代码是否符合计划规格
  2. **代码质量 Review**：实现是否足够健壮
- Review 发现问题 → 返回修复 → 再次 Review

**模式二：并行（Parallel Agents）**
- 适用于 3+ 个互相独立的问题
- 每个 Agent 接收隔离上下文，聚焦单一子系统
- 完成后合并，运行完整测试套件验证无冲突

**模型选择策略**：

| 任务类型 | 推荐模型 |
|---------|---------|
| 简单/独立任务 | 快速/低成本模型 |
| 多文件集成 | 标准模型 |
| 架构决策/Review | 最强模型 |

---

### ⑤ Test-Driven Development — TDD

**技能**：`test-driven-development`

**铁律**：Red → Green → Refactor

```
1. 写一个失败的测试（RED）
2. 运行并确认它失败
3. 写最小实现让测试通过（GREEN）
4. 重构，不改功能（REFACTOR）
5. 重复
```

**"看到失败"是核心**：
> "If you didn't watch the test fail, you don't know if it tests the right thing."

**禁止的合理化借口**：
- "太简单了不需要测"
- "先写代码，后补测试"
- "我手动测过了"

---

### ⑥ Code Review — 代码审查

**技能**：`requesting-code-review` / `receiving-code-review`

**严重级别**：

| 级别 | 含义 | 行动 |
|------|------|------|
| CRITICAL | 安全漏洞或数据丢失风险 | 必须修复才能合并 |
| HIGH | Bug 或重大质量问题 | 合并前应修复 |
| MEDIUM | 可维护性问题 | 考虑修复 |
| LOW | 风格建议 | 可选 |

---

### ⑦ Branch Completion — 完成分支

**技能**：`finishing-a-development-branch`

- 确认所有任务完成
- 所有测试通过
- Review 意见处理完毕
- 做出合并决策
- 清理 worktree

---

## 关键设计思想

每个阶段都有**强制进入条件**，不允许跳步：

- 没有批准的设计 → 不能 Plan
- 没有通过的测试基线 → 不能开始执行
- 没有失败的测试 → 不能写实现代码
- 没有验证证据 → 不能声称完成
